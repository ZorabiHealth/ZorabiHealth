## PERSON 2 — Dataflow, Dual Sync Queues & Offline Pipeline

| #   | Broken Pipe                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      | File:Line                                                                                               | Severity     |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------- | ------------ |
| 11  | **DUAL OFFLINE QUEUE SYSTEMS (architectural failure)** — Two independent queue systems with overlapping table coverage: `zh_sync_queue` (supabase.ts, 9 tables: medications, medication_logs, vendors, refill_orders, refill_order_events, voice_messages, workouts, workout_schedule, nutrition_logs) and `zh_offline_queue` (alarm-queue.ts, 2 tables: medication_logs, medications). The same action (e.g., medication log insert) can be queued in BOTH simultaneously, causing duplicate writes on reconnect. No coordination between them. | `zorabihealth/app/lib/supabase.ts:42-93` + `zorabihealth/app/lib/alarm-queue.ts:164-213`                | **Critical** |
| 12  | **Web `drainSyncQueue` uses `upsert` for ALL non-delete operations** — Even brand-new inserts use `upsert` with a client-generated UUID. If the UUID happens to collide with a stale/deleted record (e.g., from a previous sync), it silently overwrites existing data instead of failing cleanly.                                                                                                                                                                                                                                               | `zorabihealth/app/lib/supabase.ts:76`                                                                   | High         |
| 13  | **Mobile offline queue has NO `delete` operation support** — `syncOfflineQueue` (both in alarm-queue.ts and background-sync.ts) only handles `insert` and `update`. Any `delete` action queued is silently skipped during drain. Deleted items remain in Supabase.                                                                                                                                                                                                                                                                               | `zorabihealth-mobile/lib/alarm-queue.ts:260-276` + `zorabihealth-mobile/lib/background-sync.ts:128-146` | High         |
| 14  | **No conflict resolution (last-write-wins is broken)** — Two queues + two platforms with no vector clock, no timestamp comparison, no merge strategy. Web might drain `zh_sync_queue` while mobile drains `zh_offline_queue`. Final state depends on timing, NOT user intent. Example: user edits medication on mobile offline THEN deletes it on web offline → result depends on which queue drains first.                                                                                                                                      | Both sync files                                                                                         | **Critical** |
| 15  | **Medication log query has NO `user_id` filter on web** — `supabase.from("medication_logs").select("*")` fetches ALL users' medication logs. This only works in testing because the test DB has sparse data. In production, this leaks every user's log data to every other user.                                                                                                                                                                                                                                                                | `zorabihealth/app/dashboard/medications/page.tsx:153-158`                                               | **Critical** |
| 16  | **Web and mobile `buildScheduleFromRemote` are IDENTICAL DUPLICATES** — 70-line function copied verbatim between `zorabihealth/app/lib/alarm-queue.ts:87-161` and `zorabihealth-mobile/lib/alarm-queue.ts:163-232`. Any bug fix or enhancement must be applied to both separately; they will inevitably diverge.                                                                                                                                                                                                                                 | Both `alarm-queue.ts` files                                                                             | High         |
| 17  | **Background sync task on mobile has its OWN copy of `syncOfflineQueue`** — `background-sync.ts:120-148` duplicates the exact same logic from `alarm-queue.ts:252-283`. Four copies of the same queue-drain logic exist across the codebase.                                                                                                                                                                                                                                                                                                     | `zorabihealth-mobile/lib/background-sync.ts:120-148`                                                    | High         |
| 18  | **No retry limit or exponential backoff on any sync queue** — `remaining.push(item)` retries indefinitely on every `online` event. A single permanently-failed item (e.g., FK constraint violation) blocks all subsequent items in the queue forever.                                                                                                                                                                                                                                                                                            | `zorabihealth/app/lib/supabase.ts:81`                                                                   | Medium       |
| 19  | **Sync queue grows unbounded in localStorage** — No maximum size, no pruning of old/duplicate entries. `localStorage` (~5MB per origin) can fill up, causing `QuotaExceededError` on ALL localStorage writes across the app.                                                                                                                                                                                                                                                                                                                     | `zorabihealth/app/lib/supabase.ts:52-53`                                                                | Medium       |
| 20  | **Mobile `fetchMedications` on AppState change has stale closure** — `useEffect` dependency array is `[user?.id]` but the callback captures `user?.id` from initial closure. If user signs out and a different user signs in without full app restart, old user's data continues to display.                                                                                                                                                                                                                                                     | `zorabihealth-mobile/app/(tabs)/medications.tsx:157-164`                                                | Medium       |

## PERSON 3 — Notifications, Cross-Device Pairing & Real-Time Sync

| #   | Broken Pipe                                                                                                                                                                                                                                                                                                                                                           | File:Line                                                        | Severity     |
| --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- | ------------ |
| 21  | **Deepgram API key served WITHOUT AUTHENTICATION** — GET `/api/deepgram/token` returns the `DEEPGRAM_API_KEY` environment variable to ANY caller. No auth check, no rate limit, no origin validation. Anyone who discovers this endpoint gets free STT access billed to your account.                                                                                 | `zorabihealth/app/api/deepgram/token/route.ts:5-16`              | **Critical** |
| 22  | **Notification dispatch has TOCTOU race condition** — Between checking `existingDelivery` (line 90: `.eq("notification_id", notif.id).eq("device_id", device.id).maybeSingle()`) and sending the push (line 103), another cron invocation can pass the same check and send a duplicate notification. No advisory lock, no delivery idempotency key.                   | `zorabihealth/app/api/notifications/dispatch/route.ts:90-109`    | High         |
| 23  | **Expo push token expiry is NEVER handled** — `sendExpoPush` can receive an error response indicating an invalid/expired token, but the device is NEVER deactivated (`is_active = false` is never set). Dead Expo tokens accumulate in `notification_devices`, causing error logs every 5-minute cron cycle with no benefit.                                          | `zorabihealth/app/lib/notifications-transport.ts:66-97`          | High         |
| 24  | **Paired notification delivery has NO privacy filter** — When a web notification is forwarded to a paired mobile device (via `user_pairings`), ALL content (title, body, data, category, priority) is sent unconditionally. If the paired user is a different person (family caregiver), they see private health data. No opt-in for sensitive categories.            | `zorabihealth/app/api/notifications/dispatch/route.ts:62-83`     | **Critical** |
| 25  | **Mobile background sync polling LOSES notifications** — Polls every 60 minutes with `select(...).limit(10)`. If more than 10 notifications arrive in one interval, only the 10 most recent are fetched. `lastChecked` advances past the missed ones, so they are lost forever.                                                                                       | `zorabihealth-mobile/lib/background-sync.ts:196-203`             | High         |
| 26  | **`sent_via` transport tracking is lossy** — `sent_via` is stored as a JSON array per-notification. A notification targeting web + paired mobile that succeeds on web but fails on mobile is updated to `["web_push"]` only. The cron dispatcher sees `sent_via` doesn't include `"expo_push"` and retries EVERY device every 5 minutes forever.                      | `zorabihealth/app/api/notifications/dispatch/route.ts:157-160`   | Medium       |
| 27  | **Pairing code generation has TOCTOU race** — Checks uniqueness (`select ... eq("code", code).is("claimed_at", null).maybeSingle()`) THEN inserts (`insert({ code, user_id, expires_at })`). Under concurrent requests (e.g., rapid retry from mobile UI), two identical codes can be inserted. Uniqueness constraint is NOT enforced at DB level.                    | `zorabihealth/app/api/notifications/pairing/init/route.ts:16-34` | Medium       |
| 28  | **No real-time cross-device alarm cancellation** — User marks "taken" on web → mobile app does NOT receive a real-time Supabase Realtime event to cancel the local notification alarm. Mobile only refreshes alarm state on `AppState` change (when app comes to foreground) or manual pull-to-refresh. Alarm can ring on mobile 5+ minutes after being taken on web. | Web `medications/page.tsx:392` → Mobile `medications.tsx:157`    | High         |
| 29  | **VAPID public key missing has no error handling** — `useNotifications.ts` reads `NEXT_PUBLIC_VAPID_PUBLIC_KEY` and silently returns if undefined. No user-facing feedback that push notifications are unavailable due to configuration.                                                                                                                              | `zorabihealth/app/hooks/useNotifications.ts:58-63`               | Medium       |
| 30  | **Notification checkpoint file grows unbounded on mobile** — `saveCheckpoint` trims `shownIds` to last 100 entries, but the file `zh_notification_checkpoint.json` is never deleted or fully reset. Over years of use, the write accumulates metadata with no cleanup mechanism.                                                                                      | `zorabihealth-mobile/lib/background-sync.ts:286-296`             | Low          |

## PERSON 4 — Vulnerabilities, Missing Failure Test Cases & Security

| #   | Broken Pipe                                                                                                                                                                                                                                                                                                                                                                                                                                                                    | File:Line                                                                      | Severity     |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------ | ------------ |
| 31  | **`orders/status/route.ts` imports CLIENT-SIDE `supabase` (anon key) for server writes** — Line 2 imports `supabase` from `@/lib/supabase` which uses the anon key. Server-side status updates should use `supabaseAdmin` (service_role key) to bypass RLS. Writes executed with anon key will fail if RLS policies aren't configured for self-service status updates.                                                                                                         | `zorabihealth/app/api/orders/status/route.ts:2`                                | **Critical** |

A malicious client can: spam push notifications, exhaust Deepgram/Gemini quota, fill the database with fake device registrations.                                                                  | All API routes                                                                 | High         |
| 33  | **Emergency contact escalation not implemented** — `alertAfterMisses` field is stored on the Medication model and shown in UI, but there is NO backend process (cron, trigger, or webhook) that counts consecutive missed doses and escalates to the emergency contact. The entire feature is UI-only.                                                                                                                                                                         | `zorabihealth/app/lib/medications.ts:31-36` + `medications/page.tsx:1044-1058` | High         |
| 34  | **Medication `endDate` is stored but NEVER enforced** — Medications with `endDate` in the past continue to trigger alarms, appear as active in all lists, and are included in stock calculations. No cleanup cron job or dashboard warning for expired prescriptions.                                                                                                                                                                                                          | `zorabihealth/app/lib/medications.ts:27`                                       | Medium       |
| 35  | **`handleTestSMS` doesn't test SMS** — Creates a `medication_logs` row with `status: "pending"` but never calls the push notification, SMS, or Telegram pipeline. Misleading name and implementation create false confidence in the alert system.                                                                                                                                                                                                                              | `zorabihealth/app/dashboard/medications/page.tsx:463-490`                      | Low          |
| 36  | **ZERO test coverage for critical system paths** — The single test file (`doctor-edge-cases.test.ts`, 459 lines) covers only pure utility functions (date formatting, duration parsing, simple validation). NO tests exist for: sync queue drain logic, offline queue processing, notification dispatch, device pairing flow, medication CRUD operations, API route authorization, Supabase RLS policies, mobile components, background sync, or any integration/E2E scenario. | `zorabihealth/app/__tests__/doctor-edge-cases.test.ts` only                    | **Critical** |
| 37  | **Pharmacy store (`zobraipharm`) cart and orders are localStorage-only with NO cloud backup** — `zobraipharm_cart` and `zobraipharm_orders` persist only in browser localStorage. No Supabase sync, no cloud backup. If user clears browser data, switches devices, or experiences localStorage corruption, ALL orders and cart contents are permanently lost.                                                                                                                 | `zorabihealth/app/lib/pharmacy-store-data.ts:383-418`                          | High         |
| 38  | **Mobile medication save uses `local-${Date.now()}` as fallback ID** — No UUID generation on mobile. These temporary IDs (`local-1718000000000-a1b2c3`) do not conform to UUID format expected by DB constraints. When synced offline, they can cause type mismatch errors or collide with real UUIDs from the server.                                                                                                                                                         | `zorabihealth-mobile/app/(tabs)/medications.tsx:256`                           | Medium       |
| 39  | **Gemini API key exposure risk via error messages** — `/api/gemini/chat/route.ts` and `/api/gemini/meals/route.ts` proxy requests to Google Gemini API. If the upstream service returns an error with the API key in the response body, this key could be forwarded to the client. No error sanitisation layer.                                                                                                                                                                | `zorabihealth/app/api/gemini/chat/route.ts`                                    | Medium       |
| 40  | **Pharmacy vendor registration generates FAKE GPS coordinates** — `handleVendorSubmit` creates a vendor with `lat: 12.97 + Math.random() * 0.1` and `lng: 77.59 + Math.random() * 0.1`. No address geocoding, no address verification, no license validation. Vendors appear at random locations within ~10km of Bangalore centre.                                                                                                                                             | `zorabihealth/app/dashboard/pharmacy/page.tsx:584-585`                         | Medium       |
