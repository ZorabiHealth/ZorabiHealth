# ZorabiHealth — Database Memory & Reference Schema

This document serves as the master database reference for **ZorabiHealth**. It outlines the schema architecture, tables, columns, constraints, Row-Level Security (RLS) policies, triggers, custom functions, and database indexes.

---

## 1. Schema Architecture Overview

ZorabiHealth features a highly automated **Unified Entity Chain** that connects the patient, doctor, clinical record, and fulfillment pipelines. The schema flows as follows:

```mermaid
graph TD
    auth.users[auth.users] -->|1:1| user_roles[user_roles]
    auth.users -->|1:1| patient_profiles[patient_profiles]
    auth.users -->|1:1| doctor_profiles[doctor_profiles]
    auth.users -->|1:1| pharmacy_profiles[pharmacy_profiles]

    doctor_profiles -->|1:N| appointments[appointments]
    patient_profiles -->|1:N| appointments

    appointments -->|1:1 (optional)| prescriptions[prescriptions]
    doctor_profiles -->|1:N| prescriptions
    patient_profiles -->|1:N| prescriptions

    prescriptions -->|1:N| prescription_items[prescription_items]
    prescriptions -->|1:N| medications[medications]

    medications -->|1:N| medication_logs[medication_logs]
    medications -->|1:N (optional)| refill_orders[refill_orders]

    prescriptions -->|1:N (optional)| orders[orders]
    pharmacy_profiles -->|1:N| orders
    orders -->|1:N| order_events[order_events]

    pharmacy_profiles -->|1:N| pharmacy_inventory[pharmacy_inventory]
    drug_catalog[drug_catalog] -->|1:N| pharmacy_inventory
    drug_catalog -->|1:N (optional)| prescription_items

    patient_profiles -->|1:N| vital_signs[vital_signs]
    patient_profiles -->|1:N| symptom_logs[symptom_logs]

    auth.users -->|1:N| store_orders[store_orders]
    store_orders -->|1:N| store_order_items[store_order_items]
    store_orders -->|1:N| store_order_events[store_order_events]
```

## 2. Table Definitions Reference

### 2.1 User Management & Profiles

#### `user_roles`

Tracks the role of each user registered in the system.

- `user_id` (uuid, primary key): References `auth.users(id)`
- `role` (text, not null): Check constraint `('patient', 'doctor', 'pharmacy_vendor')`
- `created_at` (timestamptz, default `now()`)

#### `patient_profiles`

Allows doctor profiles to manage patient records, even if patients have not registered an auth account yet.

- `id` (uuid, primary key): Matches `auth.users.id` if registered, or generated by doctor.
- `full_name` (text, not null)
- `email` (text, optional)
- `phone` (text, optional)
- `created_by` (uuid): References `doctor_profiles(id)`
- `created_at` (timestamptz, default `now()`)

#### `doctor_profiles`

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `user_id` (uuid, unique): References `auth.users(id)`
- `license_number` (text, unique, not null)
- `specialization` (text, not null)
- `qualification` (text, not null)
- `hospital_affiliation` (text, optional)
- `consultation_fee` (numeric(10,2), default `0`)
- `is_verified` (boolean, default `false`)
- `doctor_name` (text, optional): Backfilled/seeded for displaying names.
- `created_at` (timestamptz, default `now()`)
- `updated_at` (timestamptz, default `now()`)

#### `pharmacy_profiles`

Replaces legacy `vendors` in modern workflows.

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `user_id` (uuid, unique): References `auth.users(id)`
- `business_name` (text, not null)
- `license_number` (text, unique, not null)
- `address` (text, not null)
- `pincode` (text, not null)
- `lat` (double precision, not null)
- `lng` (double precision, not null)
- `phone` (text, optional)
- `operating_hours` (text, default `'Mon-Sat 09:00-21:00'`)
- `delivery_radius_km` (double precision, default `7.0`)
- `is_verified` (boolean, default `false`)
- `is_active` (boolean, default `true`)
- `rating` (double precision, default `4.5`)
- `created_at` (timestamptz, default `now()`)
- `updated_at` (timestamptz, default `now()`)

### 2.2 Clinical & Consultation Core

#### `appointments`

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `tracking_id` (text, unique): Auto-generated prefix `ZR-APPT-YYYYMMDD-[RANDOM-HEX]`
- `patient_id` (uuid, not null): References `patient_profiles(id)`
- `doctor_id` (uuid, not null): References `doctor_profiles(id)`
- `scheduled_date` (date, not null)
- `scheduled_time` (text, not null)
- `status` (text, default `'scheduled'`): Check `('scheduled', 'completed', 'cancelled')`
- `type` (text, default `'In-Person'`): e.g., `'In-Person'`, `'Video Consultation'`
- `notes` (text, optional)
- `created_at` (timestamptz, default `now()`)
- `updated_at` (timestamptz, default `now()`)

#### `prescriptions`

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `tracking_id` (text, unique): Auto-generated prefix `ZR-RX-YYYYMMDD-[RANDOM-HEX]`
- `appointment_id` (uuid, optional): References `appointments(id)`
- `doctor_id` (uuid, not null): References `doctor_profiles(id)`
- `patient_id` (uuid, not null): References `patient_profiles(id)`
- `diagnosis` (text, optional)
- `notes` (text, optional)
- `status` (text, default `'draft'`): Check `('draft', 'active', 'filled', 'partially_filled', 'expired', 'cancelled')`
- `ai_assisted` (boolean, default `false`)
- `created_at` (timestamptz, default `now()`)
- `updated_at` (timestamptz, default `now()`)

#### `prescription_items`

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `prescription_id` (uuid, not null): References `prescriptions(id)`
- `drug_id` (uuid, optional): References `drug_catalog(id)`
- `drug_name` (text, not null)
- `dosage` (text, not null)
- `frequency` (text, not null)
- `duration` (text, not null)
- `quantity` (integer, not null): Check `(quantity > 0)`
- `notes` (text, optional)
- `created_at` (timestamptz, default `now()`)

#### `prescription_documents`

Stores generated PDF records of prescriptions.

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `prescription_id` (uuid, not null): References `prescriptions(id)`
- `storage_path` (text, not null): The path to the PDF file in Supabase storage bucket `prescription_pdfs`
- `created_at` (timestamptz, default `now()`)

### 2.3 Catalog & E-Commerce / Pharmacy Inventory

#### `drug_catalog`

Master database of medicines.

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `name` (text, not null)
- `generic_name` (text, optional)
- `category` (text, default `'General'`)
- `manufacturer` (text, optional)
- `created_at` (timestamptz, default `now()`)
- Unique constraint on `(name, manufacturer)`

#### `pharmacy_inventory`

Stocks and pricing configurations for each pharmacy.

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `pharmacy_id` (uuid, not null): References `pharmacy_profiles(id)`
- `drug_id` (uuid, not null): References `drug_catalog(id)`
- `price_per_unit` (numeric(10,2), not null): Check `(price_per_unit >= 0)`
- `stock` (integer, default `0`): Check `(stock >= 0)`
- `is_available` (boolean, default `true`)
- `updated_at` (timestamptz, default `now()`)
- Unique constraint on `(pharmacy_id, drug_id)`

#### `orders`

B2B or Direct Prescription Orders dispatched to pharmacies.

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `prescription_id` (uuid, optional): References `prescriptions(id)`
- `appointment_id` (uuid, optional): References `appointments(id)`
- `pharmacy_id` (uuid, not null): References `pharmacy_profiles(id)`
- `patient_id` (uuid, not null): References `auth.users(id)`
- `status` (text, default `'pending'`): Check `('pending', 'confirmed', 'preparing', 'dispatched', 'delivered', 'cancelled')`
- `total_amount` (numeric(10,2), not null)
- `delivery_address` (text, not null)
- `tracking_id` (text, unique, optional)
- `created_at` (timestamptz, default `now()`)
- `updated_at` (timestamptz, default `now()`)

#### `order_events`

- `id` (uuid, primary key, default `gen_random_uuid()`)
- `order_id` (uuid, not null): References `orders(id)`
- `status` (text, not null)
- `note` (text, optional)

#### `store_orders`

E-commerce retail orders from **ZorabiPharm**.

- `user_id` (uuid, not null): References `auth.users(id)`

- `customer_name` (text, not null)

#### `store_order_items`

## 3. Row-Level Security (RLS) Policies

Every table enforces Row-Level Security. General access controls are as follows:

| Table                | SELECT                                                     | INSERT                                  | UPDATE / DELETE                         |
| :------------------- | :--------------------------------------------------------- | :-------------------------------------- | :-------------------------------------- |
| `user_roles`         | Only own (`user_id = auth.uid()`)                          | Only own                                | Only own                                |
| `patient_profiles`   | Authenticated users (public directory)                     | Creators (Doctor matching `created_by`) | Creators (Doctor matching `created_by`) |
| `doctor_profiles`    | Authenticated users (public directory)                     | Only own (`user_id = auth.uid()`)       | Only own                                |
| `pharmacy_profiles`  | Authenticated users (public directory)                     | Only own (`user_id = auth.uid()`)       | Only own                                |
| `appointments`       | Doctor or Patient linked to appointment                    | Doctor or Patient linked to appointment | Doctor or Patient linked to appointment |
| `prescriptions`      | Doctor linked, or Patient (`patient_id = auth.uid()`)      | Doctor linked to prescription           | Doctor linked to prescription           |
| `prescription_items` | Linked doctor or patient                                   | Linked doctor only                      | Linked doctor only                      |
| `orders`             | Patient (`patient_id = auth.uid()`) or Pharmacy            | Pharmacy linked to order                | Pharmacy linked to order                |
| `store_orders`       | Only own (`user_id = auth.uid()`)                          | Only own                                | Only own                                |
| `medications`        | Only own (`user_id = auth.uid()`)                          | Only own                                | Only own                                |
| `medication_logs`    | Owned via medications (`medications.user_id = auth.uid()`) | Owned via medications                   | Owned via medications                   |
| `vital_signs`        | Doctor linked, or Patient (`patient_id` matches profile)   | Doctor linked to record                 | Doctor linked to record                 |
| `symptom_logs`       | Only own (`user_id = auth.uid()`)                          | Only own                                | Only own                                |
| `wearable_alarms`    | Only own (`user_id = auth.uid()`)                          | Only own                                | Only own                                |
| `sleep_sessions`     | Only own (`user_id = auth.uid()`)                          | Only own                                | Only own                                |
| `daily_steps`        | Only own (`user_id = auth.uid()`)                          | Only own                                | Only own                                |

## 4. Database Functions & Triggers

### 4.1 Modtime Auto-Update Trigger

Standard trigger applied to `doctor_profiles`, `pharmacy_profiles`, `appointments`, `prescriptions`, `orders`, `medications`, `refill_orders`, `wearable_alarms`, and `daily_steps`.

- **Function**: `update_modified_column()`
- **Action**: Sets `new.updated_at = now()` before update on each row.

### 4.2 Auto-Generating Tracking IDs

Ensures tracking IDs are assigned on record creation.

- **Prescription Trigger**: `trg_set_prescription_tracking_id()` before insert on `prescriptions` generates IDs with format `ZR-RX-YYYYMMDD-[6-CHAR-HEX]`.
- **Appointment Trigger**: `trg_set_appointment_tracking_id()` before insert on `appointments` generates IDs with format `ZR-APPT-YYYYMMDD-[6-CHAR-HEX]`.

### 4.3 Automated Medication Creation on Active Prescription

Automatically schedules patient medication intake upon prescription completion.

- **Trigger**: `prescription_fill_to_medications` on update of `status` on `prescriptions` when status changes to `'active'`.
- **Logic**: Parses prescription items, matches frequency strings (e.g. `'twice'` -> `'twice_daily'`, scheduled times `'{08:00,20:00}'`), sets default stock and refill levels, and creates corresponding entries in the `medications` table.

### 4.4 Automated Refill Linking

- **Trigger**: `refill_order_link_prescription` before insert on `refill_orders`.
- **Logic**: Automatically checks if `medication_id` has an associated `prescription_id` and populates the field to maintain auditability.

### 4.5 Trace Entity Analysis

- **Function**: `trace_entity(p_entity_id uuid)`
- **Returns**: A table displaying the active status and detailed metadata of a record along the entire entity chain (Patient details, Appointments, Prescriptions, Medications, Orders). Serves as an analytical tool for checking patient adherence.

## 5. Performance & Optimization Indexes

Critical indexes created for rapid retrieval of records and search optimization:

- **`idx_patient_profiles_name`**: Index on `patient_profiles(full_name)` for patient directory listings.
- **`idx_doctor_profiles_specialization`**: Index on `doctor_profiles(specialization)` for doctor searches.
- **`idx_appointments_lookup`**: Indexes on `appointments(patient_id, scheduled_date)` and `appointments(doctor_id, scheduled_date)` for calendars.
- **`idx_medications_user_active`**: Index on `medications(user_id, is_active)` to filter active schedules.
- **`idx_medication_logs_lookup`**: Index on `medication_logs(medication_id, scheduled_at, status)` to query compliance logs.
- **`idx_daily_steps_lookup`**: Index on `daily_steps(user_id, date desc)`.
- **`idx_sleep_sessions_lookup`**: Index on `sleep_sessions(user_id, date desc)`.
- **`idx_voice_messages_text_search`**: GIN index on voice message transcripts using `to_tsvector` for full-text search capability.
